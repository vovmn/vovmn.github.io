# Иммунолаб

## Архитектурный подход

Используется чистая архитектура. Все решение делится на несколько слоев:  
- **Domain** (домен). Содержит сущности, которые соответствуют структурам в БД, а также репозиторий, определяющий взаимодействия с этими сущностями на уровне БД.
Данный слой является самым низким в иерархии и ни от кого не зависит.
- **Infrastructure** (инфраструктура). Определяет взаимодействия с внешними сервисами, например с БД. Подслой _Persistence_ хранит информацию о том, как именно 
хранятся данные. Например, у нас это PostgreSQL. Данный слой зависит от Domain.
- **Application** (приложение). Содержит определения бизнес-логики приложения. Здесь расположены обработчики комманд и запросов. Зависит от _Infrastructure_.
- **Presentation** (презентация). Содержит интерфейсы для взаимодействия конечного пользователя с нашим приложением. У нас для этого есть Web API приложение (сервер) 
и Blazor приложение (клиент).

Также используется паттерн CQRS - Command and Query Segregation Principle (принцип разделения комманд и запросов). 
Операции делятся на комманды - то, что меняет состояние системы, и на запросы - то, что не меняет состояние системы. Примеры комманд: создание, удаление и обновление пациентов.
Примеры запросов: получение списка пациентов.

## Структура решения

Проект **Immunitas.Domain**. Реализует слой *Domain* чистой архитектуры.
  - Entities. Сущности, соответствующие структурам в БД.

Проект **Immunitas.Infrastructure**. Реализует слой *Infrastructure* чистой архитектуры.
- Persistence. Содержит информацию о том, как хранятся данные в конкретном провайдере (БД).
  - Configs. Содержит настройки сущностей. В частности переопределения того, как C# классы представляются в БД.
  - Conventions. Специальные конвенции, которые должны соблюдаться при сопоставлении C# классов структурам в БД.
Например, задаем, что названия классов в _CamelCase_ должны соответствовать названиям таблиц в _snake_case_.
  - Migrations. Миграции - описания того, как конкретно должны создаваться таблицы в БД. Генерируются автоматически на основании настроек сущностей
  - Repositories. Реализация интерфейса репозитория по взаимодействию с БД.
 
Проект **Immunitas.Application**. Реализует слой *Application* чистой архитектуры.
- BloodTests, Patients и т.д. Каждая папка соответствует доменной сущности.
  - Queries. Запросы и их обработчики на получение сущностей, которые не меняют состояние системы.
  - Commands. Команды и их обработчики на создание, обновление и удаление сущностей. В общем, всё то, что меняет состояние системы.
- DTOs. Data Transfer Objects - структуры данных, для передачи их между БД и конечным пользователем.

> [!Important]
> Модельки из **Application**, то есть команды, запросы и DTO **автоматически** переносятся на клиентское приложения во время сборки приложения 
> при помощи генератора и маркера `SharedContract`.

Проект **Immunitas.Portal.Bff**. Реализует слой Presentation чистой архитектуры. Представляет собой Web API приложение.
- Controllers. Содержит контроллеры - обработчики HTTP-запросов. ASP.NET автоматически парсит запрос и вызывает конкретный обработчик для этого запроса.
- ExceptionHandlers. Содержит обработчики исключений, возникающий в ходе работе приложения. Например, можно перехватить исключение `PatientNotFound` и вернуть
ответ со статусом 404 Not Found. Обработчики могут выстраиваться в цепочки, что позволяет гибко обрабатывать разные исключения.

Проект **Immunitas.Portal.UI**. Также реализует слой *Presentation* чистой архитектуры. Представляет собой клиентское Blazor приложение.
- wwwroot. Содержит статические файлы, такие как html, css, js и т.д.
- Layout. Содержит шаблоны, в которые уже затем помещаются другие компоненты - тело страницы.
- Pages. Содержит компоненты, соответствующие страницам в браузере (условно).

Проект **Immunitas.Portal.UI.ApiProxies**. Вспомогательный проект для клиентского приложения, содержащий классы для осуществления HTTP запросов к серверу и 
модельки для сериализации запросов к серверу и десериализации ответов от сервера. Как раз сюда попадают модельки, помеченные `SharedContract` в **Immunitas.Application**.
В будущем, возможно, доработаю генератор так, чтобы и классы для осуществления HTTP запросов создавались автоматически.
